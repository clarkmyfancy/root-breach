# Root Breach V2 Canonical Requirements

Status: Authoritative `WHAT` specification for V2 behavior and content.  
Audience: Any implementer working in this repository.  
Scope: Product/requirements contract, not file-by-file implementation instructions.

Normative language:
- `MUST` is mandatory for MVP acceptance.
- `SHOULD` is strongly recommended.
- `MAY` is optional or post-MVP.

## 1) Canonical Game Model (Source of Truth)

All entities below are required and must exist as explicit concepts.

### 1.1 `SiteBlueprint`
Represents reusable site topology and operational scope.

```ts

  id: string;                  // Stable site id
  levelId: string;             // Back-reference to level data (legacy bridge)

  authEndpoints: string[];     // Contract-addressable auth endpoint ids
```

### 1.11 Runtime vs Persisted vs Visibility
- Runtime only:
  - `MissionRuntimeState`
  - `NodeRuntimeState`
  - live `TraceState`
  - live `AttributionResult`
  - per-run `MissionEvent` stream
- Persisted:
  - `CampaignState`
  - scripts by contract
  - best scripts by contract
  - contract completion/unlock state
  - save version + settings
- Player-visible (must be inspectable):
  - mission phase, objective status, trace, alarm, tick

  - attribution output (actor/confidence/reasons)
  - debrief pass/fail by rule

## 2) Mission Lifecycle Contract

Required phases:
- `PLANNING`: briefing, intel review, script edit/compile, no tick execution.
- `OBJECTIVE`: mission live; objective actions executed.
- `CLEANUP`: objective complete, counter-forensics/framing window active.
- `COMPLETE`: terminal success.
- `FAILED`: terminal failure.

Allowed actions by phase:
- `PLANNING`: UI/apps, scripting, compile, run launch.
- `OBJECTIVE`: all non-cleanup commands; cleanup-only commands are illegal.
- `CLEANUP`: all commands allowed; cleanup rules are evaluated continuously.
- `COMPLETE` and `FAILED`: read-only replay/debrief.

Phase transitions:
- `PLANNING -> OBJECTIVE`: run starts at tick 0.
- `OBJECTIVE -> CLEANUP`: objective complete and contract requires cleanup/frame/forced-detection checks.
- `OBJECTIVE -> COMPLETE`: objective complete and no cleanup required.
- `OBJECTIVE -> FAILED`: hard fail (trace max, tick limit, fatal rule violation, lethal failure where enabled).
- `CLEANUP -> COMPLETE`: all cleanup and attribution rules satisfied before deadline.
- `CLEANUP -> FAILED`: deadline reached or cleanup/attribution rule violated.

Cleanup requirements:
- Cleanup is required when any of:
  - `requireNoTrace = true`
  - `allowFrameTarget = true`
  - `forcedDetection = true`
- Cleanup may be skipped only when all above are false.

Contract outcome modes:
- `Ghost`: objective success + clean/low-trace + no operator attribution.
- `Frame`: objective success + required target attribution achieved.
- `Loud Diversion`: detection/alarm allowed or required, but operator attribution must fail.

## 3) Final Win/Fail Contract (Authoritative)

### 3.1 Success Conditions
Mission succeeds only when all are true:
- Objective conditions for `objectiveType` pass.
- If cleanup required, cleanup checks pass before deadline.
- Trace rules pass:
  - hard fail at `trace >= 100`.
  - for `requireNoTrace`, clean threshold must be met (`trace < 40` at closure).
- Attribution rules pass:
  - if framing required, `suspectedActor` matches target and `confidence >= 0.60`.
  - if anti-attribution rule exists, `suspectedActor !== OPERATOR`.
- Contract-specific rules pass.

### 3.2 Failure Conditions
Mission fails when any true:
- `trace >= 100`.
- tick limit reached before valid completion.
- objective not completed.
- cleanup required but not completed in window.
- unsanitized operator evidence violates contract rules.
- framing required but target mismatch or insufficient confidence.
- forced-detection required but no detection occurred.
- explicit contract rule violations.

### 3.3 Contract Exceptions
- `forcedDetection = true` disables "no detection" assumptions and requires at least one valid detection event.
- `allowFrameTarget = true` requires evidence-backed frame outcome, not just setting a flag.
- `requireNoTrace = true` enforces strict trace/evidence closure.

### 3.4 Exfil Definition (No Physical Path Dependency)
- Exfil is a mission-state closure condition, not mandatory physical route traversal.
- Legacy path traversal MAY exist for flavor/testing, but `reach exit` MUST NOT be the universal success requirement.

## 4) Tick-Order Contract (Determinism Critical)

Each mission tick MUST execute in this order:
1. Execute scheduled script commands.
2. Apply command side effects (state changes, objective flags, evidence writes).
3. Update timed device state (camera timers, temporary overrides).
4. Move autonomous systems (drones or equivalent).
5. Evaluate sensor detection.
6. Apply alarm escalation/lockdown rules.
7. Update trace (sources, deltas, thresholds, lock state).
8. Resolve combat/security response (if enabled by contract).
9. Update movement/position systems (if enabled by mission mode).
10. Check objective completion.
11. Check cleanup/forensics completion when in cleanup phase.
12. Evaluate terminal win/fail state.
13. Advance tick and emit frame snapshot.

Additional tick rules:
- Evidence writes happen at command/system action point, not delayed post-processing.
- Phase transitions are emitted as events in the same tick they occur.
- Same initial state + script + seed MUST produce identical events and outcome.

## 6) Trace System Requirements

Trace model:
- Range: `0..100`.
- Hard thresholds: `25`, `50`, `75`, `100`.
- Hard mission fail at `100`.

Required contributors:
- persistent connection baseline
- active global heat modifier
- live detection state
- alarm state pressure
- noisy command execution pressure
- cleanup-phase unsanitized operator evidence pressure

Required reducers:
- relay routing effects
- decoy burst absorption
- explicit spoof action effects

Contract interactions:
- `requireNoTrace`: strict clean closure; trace threshold check required.
- `forcedDetection`: detection required but trace may still fail mission.
- trace failure MUST affect debrief and campaign heat outcome.

Campaign interaction:
- Contract failure MAY increase global heat by contract-defined amount.
- Global heat MUST influence future trace growth.

Player visibility:
- UI MUST show current trace, delta per tick, and top source reasons.

## 7) Evidence and Forensics Requirements

Required surfaces:
- `AUTH`
- `DEVICE`
- `FILE_AUDIT`
- `PROCESS`
- `NETFLOW`
- `ALARM`

Evidence state semantics:
- `scrubbed = true`: record exists historically but excluded from active attribution checks.
- `forged = true`: operator-authored evidence crafted to influence attribution.
- `hidden = true`: not shown by default unless revealed by probe/forensics view.
- `attributedTo`: actor currently associated with evidence for scoring.

System requirements:
- Every meaningful action MUST either generate evidence or explicitly declare no evidence.
- Cleanup/framing outcomes MUST depend on evidence state, not only objective flags.
- Frame success MUST require evidence support.
- Forensics panel MUST show evidence by surface and state (scrubbed/forged/hidden/attributed).

Optional later rule:
- Excessive scrubbing MAY create suspicion as separate evidence.

## 8) Attribution Model Requirements

Candidate actors:
- `OPERATOR`
- `UNKNOWN`
- Contract frame targets (for example `RIVAL_OPS`, `FACILITY_MAINT`, `CONTRACTOR_ZETA`)
- Any spoofed identities introduced by tools

Scoring inputs:
- unsanitized evidence count and severity by surface
- forged evidence markers
- spoofed identities and relay signatures
- frame target rule hints

Required deterministic output:
- `suspectedActor`
- `confidence` (`0..1`)
- top reasons list (at least 2 when available)

Required rule checks:
- Contracts can require:
  - `suspectedActor !== OPERATOR`
  - `suspectedActor === targetFrameIdentity`
  - optional minimum confidence threshold

Attribution MUST be visible both:
- live in mission (Forensics Snapshot)
- terminal in debrief

## 10) Removed/Deprecated Core Assumptions

- physical path traversal as primary puzzle axis

## 15) Debrief Requirements

Debrief MUST show:
- objective result (pass/fail)
- cleanup result (pass/fail/na)
- trace result (final value + threshold status)
- attribution result (actor + confidence + reasons)
- contract rule checks (frame match, no-trace, forced-detection, etc.)
- payout delta
- reputation delta
- heat delta
- explicit failure explanation bullets ("why exposed/why failed")

## 16) Non-Functional Requirements

Must remain true:
- deterministic simulation (`same script + same state + same seed = same output`)
- explainable failures (no opaque fail conditions)
- multiple valid solutions per contract where design allows
- diegetic but readable UI
- pseudo-real systems only (no real-world exploit simulation)

## 17) Acceptance Criteria (`Done Means...`)

- A mission can succeed without mandatory physical path traversal.
- "Reach exit" is not the universal default win condition.
- Site Monitor is primary mission play surface.
- At least one contract requires cleanup to pass.
- At least one contract requires successful framing to pass.
- Forensics Snapshot shows live attribution during mission.
- Debrief explains failure in terms of trace/evidence/attribution.
- Tool ownership blocks illegal commands at compile time.
- Required contract tools block deployment in production mode.
- Simulation remains deterministic across reruns.
